# 온톨로지 구성 및 데이터 파이프라인 방안 (Ontology & Pipeline)

본 문서는 `kosLINK AI` 서비스의 핵심 자산인 **3-Layer Value Chain Ontology**의 정의와 한계점을 다루고, 공공 데이터 및 API를 연동하여 온톨로지 업데이트를 **자동화 및 실시간 동기화**하는 데이터 파이프라인 구현 아키텍처를 제시합니다.

---

## 1. 3-Layer 온톨로지 스키마 정의 및 모델 구조

온톨로지 구조는 시장을 대표하는 거시 트렌드부터 미시 기업 정보까지 3단계로 연결됩니다.

```
[Layer 1: 테마 분야 (Theme)]
          │  (소속됨)
          ▼
[Layer 2: 밸류체인 역할 (Role/Technology)]
          │  (공급/계약 관계 - SUPPLY_TO)
          ▼
[Layer 3: 상장 종목 (Stock)]  <-- {체급 속성: 대형주/중형주/소형주}
```

### ① 노드 (Node) 구성
- **테마 분야 Node**: 시장 지배적 트렌드 (예: `온디바이스 AI`)
- **밸류체인 역할 Node**: 생태계 내 실질적 공정 및 기술 레이어 (예: `칩 제조사`, `반도체 검사부품`, `반도체 IP 설계`)
- **상장 종목 Node**: 거래되는 개별 기업 주식 (예: `삼성전자`, `리노공업`, `ISC`, `오픈엣지테크놀로지`)
  - *핵심 속성*: 시가총액 기반 체급 태그(`대형주`/`중형주`/`소형주`)를 노드 속성에 매핑하여 비중 연산의 기틀로 사용.

### ② 엣지 (Edge) 구성
- **소속됨 (Belongs To)**: `테마 분야` ➔ `밸류체인 역할` 관계.
- **공급/계약 관계 (SUPPLY_TO)**: `밸류체인 역할` ➔ `상장 종목` 및 **`상장 종목` ➔ `상장 종목`** 간의 공급선.
  - *핵심 속성*: 관계선 위에 계약금액이나 단순 코드가 아닌, **DART 공시/뉴스 기반의 팩트 근거 텍스트(설명)**를 속성으로 기재. (예: `금감원 DART: 초미세 검사 핀 독점 확인`)

---

## 2. 공공 데이터 및 금융 API를 활용한 데이터 수집 자동화

정적 온톨로지를 탈피하여, 공공 API를 활용해 아래와 같은 기초 데이터를 자동 수집 및 동기화할 수 있습니다.

### ① 기업 및 상장 정보 수집 (KRX 및 공공 데이터)
- **활용 API**: 공공데이터포털(data.go.kr)의 **'한국거래소(KRX) 상장종목 정보 API'** 및 **'예탁결제원 주식정보 API'**
- **수집 정보**: 종목명, 주식 단축코드, 실시간 시가총액 정보.
- **자동화 로직**:
  - 매일 장 마감 후 스케줄러(예: Airflow)를 구동해 상장 종목의 시가총액 데이터를 조회.
  - 시가총액 규모에 따라 대형주(상위 1~100위), 중형주(101~300위), 소형주(그 외)로 분류하는 **{체급} 속성을 노드에 자동으로 갱신**.

### ② 공시 데이터 트래킹 (금융감독원 Open DART API)
- **활용 API**: **'금융감독원 Open DART API'** (개발자 전용 인증키 필요)
- **수집 정보**:
  - '주요사항보고서' (단일판매·공급계약체결, 지분 취득 등 계약 관계 적출용 공시)
  - '정기공시' (분기/반기/사업보고서)
- **자동화 로직**: 1시간 주기로 DART 신규 공시 접수 현황 API를 호출하여, 반도체 기업군 및 계약 금액 기준치를 초과하는 공급 계약 공시를 트래킹 및 수집.

## 3. LLM 기반 실시간 온톨로지 업데이트 파이프라인 구현 방안

신규 공급 계약 공시가 업로드되면, LLM을 활용해 문서를 해석하고 온톨로지 그래프(Graph DB)에 실시간으로 반영하는 파이프라인 아키텍처입니다.

```
[DART RSS / API] ──(신규 공시 감지)──> [텍스트 추출 및 전처리]
                                                │
                                                ▼
[Graph DB (Neo4j)] <──(인입/승인)── [Human-in-the-Loop Curation] <── [LLM 개체명/관계 추출 (JSON)]
```

### [Step 1] 공시 모니터링 및 트리거
- DART 오픈 API를 통해 등록되는 공급망 계약 및 파트너 계약 공시를 감지하여 파이프라인을 기동합니다.

### [Step 2] LLM 기반 정보 추출 (Gemini Structured Outputs 활용)
- 공시 본문(PDF/HTML 텍스트)을 LLM에 전달하고, 구조화된 JSON Schema 형태로 개체(Entity)와 공급 관계(SUPPLY_TO) 정보를 추출합니다.
- **LLM 프롬프트 가이드**:
  ```json
  {
    "supplier": "공급 기업명",
    "buyer": "수요처 기업명",
    "product": "공급 장비 및 부품 명칭",
    "contract_amount": "계약 총액",
    "fact_source": "금감원 DART 공시(2026-XX-XX)"
  }
  ```

### [Step 3] 개체 정규화 및 그래프 매핑 (Entity Resolution)
- 추출된 기업명(`supplier`, `buyer`)을 한국거래소 상장사 사전 데이터와 매핑하여 Ticker 코드 또는 UUID를 획득합니다. (예: '리노공업(주)' ➔ `S_LNO`)
- 만약 수요처(`buyer`)가 비상장 대기업 또는 해외 기업(예: Apple, TSMC)인 경우 신규 노드로 분류 등록합니다.

### [Step 4] Human-in-the-Loop (HITL) Curation (보안 검증 레이어)
- **중요성**: LLM의 환각(Hallucination)으로 인해 잘못된 공급 관계선이 생성되거나 경쟁사 간에 계약 관계가 오기입되는 것을 방지합니다.
- **방안**: 파이프라인에서 추출한 JSON 변경 이력을 임시 Staging 테이블에 적재한 뒤, **내부 리서처/보안 담당자(인간)의 대시보드 승인 버튼을 통해 최종 배포용 Graph DB(Neo4j 등)에 갱신**하도록 설계합니다.

---

## 4. 기술적 한계 및 추후 개선점

- **LLM 개체 식별 에러**: 유사한 한글 기업명이 많아 텍스트 매핑 시 혼선이 올 수 있으므로, 사업자등록번호 및 법인등록번호 기반으로 매칭하는 로직 보완이 필요합니다.
- **공시 비공개(비밀유지 계약) 장벽**: 글로벌 빅테크 공급의 경우 계약 당사자가 공시 상에서 '영업비밀'로 마스킹되는 경우가 80% 이상입니다. 이를 해소하기 위해 공시뿐만 아니라 **특허 출원 데이터** 및 **관세청 수출입 내역 데이터**를 교차 분석하는 온톨로지 확장 파이프라인이 보완되어야 합니다.
